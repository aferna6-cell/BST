ECE 2230 - Machine Problem 5
TEST PLAN

1. Overview of testing strategy
-------------------------------
The bst.c module is exercised in three layers:

  (a) Compile-time validation. The Makefile passes -DVALIDATE so that every
      call to bst_insert ends with bst_debug_validate(T). That function
      verifies BST ordering on every insert, asserts that the cached size
      matches the visited node count, and (for AVL trees) walks the tree
      asserting that |height(left) - height(right)| <= 1 at every node.
      Any structural bug therefore aborts the program at the offending
      operation.

  (b) Unit drivers (-u 0 .. -u 7) in lab5.c. These are small deterministic
      tests that hit specific structural cases (duplicate replacement,
      LL/RR/LR/RL rotations, skewed input, mixed input, plus a custom
      stress test). Each test is run twice, once with -f bst and once with
      -f avl, so we see how the same key sequence is shaped by each policy.

  (c) Access drivers (-o, -r, -p) at -w 16 (65,535 nodes) and
      -t 100,000 random lookups. These drive the implementation at scale
      and produce the internal-path-length numbers used in the
      Performance Summary.

  (d) Memory-leak verification with valgrind on a representative subset
      of the runs above.

The build command used throughout is:
    make clean && make
which compiles with `-g -Wall -DVALIDATE` and links against -lm.


2. Commands used
----------------

  Build:

  Unit drivers (run for both BST and AVL):

  Access drivers (small, with tree print):
    ./lab5 -o -w 5 -t 0 -v -f bst

3. Requirements coverage map
----------------------------

  * Correct BST ordering        -- every insert path triggers
                                   bst_debug_validate_rec, which fails
                                   any out-of-range key. Every unit test
                                   ends with "Validating tree...passed".

  * Correct AVL balancing       -- VALIDATE-mode runs call rec_height
                                   after every AVL insert, which asserts
                                   the balance factor at every node is
                                   in {-1, 0, +1}. Tests -u 1..4 verify
                                   each of LL, RR, LR, RL rotations
                                   directly; -u 5 verifies that a fully
                                   skewed input under AVL flattens to a
                                   balanced tree of height 3 instead of
                                   height 6.

  * Correct duplicate handling  -- bst_insert returns 0 on a duplicate
                                   and the data block is freed and
                                   replaced. Tested by -u 0 (BST) and
                                   -u 0 (AVL); also exercised inside
                                   -u 7 (the second insert of key 50).
                                   bst_size remains unchanged across the
                                   replacement (verified in unit output).

  * Correct tree size           -- every unit driver prints
                                   "Tree size = N" using bst_size, and
                                   the access driver asserts
                                   size == 2^Levels - 1 after building
                                   the test tree, then asserts the size
                                   is unchanged after the lookups.

  * Correct internal path len   -- printed by every unit and access run.
                                   For -u 6 (a perfectly balanced 15-node
                                   tree built bottom-up by 50,30,70,...)
                                   the IPL is 34, which matches the
                                   theoretical IPL of the unique
                                   complete binary tree on 15 nodes
                                   (1*2 + 2*4 + 3*8 = 34). For -o -w 16
                                   the IPL is 917,506, equal to the
                                   minimum IPL for 65,535 nodes
                                   (sum_{k=0..15} k*2^k ).

  * No memory leaks             -- valgrind reports
                                       in use at exit: 0 bytes in 0 blocks
                                       All heap blocks were freed
                                       ERROR SUMMARY: 0 errors
                                   on every run listed in section 2.

----------------------------------------

    Tree after inserts:
              30-1
       20-2
              10-1
      /-20-\
     10    30
    Validating tree...passed

RR rotation (insert 10, 20, 30 with -f avl) produces the same shape.
LR rotation (insert 30, 10, 20 with -f avl) produces the same shape.

For comparison, the same sequences with -f bst leave the keys in their
insertion order (height 3, IPL 3) instead of being rebalanced. This

4.8 Memory-leak verification
----------------------------------------

For every command listed in section 2, valgrind reports:

    in use at exit: 0 bytes in 0 blocks
    All heap blocks were freed -- no leaks are possible

For example, ./lab5 -u 6 -f avl makes 32 allocations and 32 frees;

5. Conclusion
-------------
The implementation passes BST ordering and AVL balance checks (asserted
by the supplied bst_debug_validate after every insert), correctly
replaces duplicates without changing tree size, reports correct size and
internal path length on every unit test, and shows zero leaks under
valgrind across all tested workloads.
